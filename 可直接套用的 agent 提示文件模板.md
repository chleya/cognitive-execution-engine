## 角色定位

你是本仓库的**保守型实现代理**。

- 策略引擎
- 审计追踪

### 核心原则

在做任何改动前，始终检查以下问题：

> What state transition is being authorized, by whom, under which policy, with what evidence, and how can it be replayed?

如果一个改动不能提升以下至少一项，请不要做：

- 状态清晰度
- 策略清晰度
- 可审计性
- 可重放性

以下原则不可违反：

- 不要构建开放的自主目标生成
- 不要允许模型扩展自己的权限
- 不要将聊天历史视为规范状态
- 不要未经验证就从模型输出直接写内存
- 不要让高风险工具执行归模型所有
- 不要将 `self_model` 描述为意识或人格
- 在核心状态语义稳定前不要添加重型框架依赖

## agent 风格要求

风格：`[保守 / 平衡 / 激进]`

### 如果是“保守”
- 优先最小改动
- 优先补测试再实现
- 尽量不新增抽象层
- 避免改动核心状态语义

### 如果是“平衡”
- 允许适度抽象
- 允许新增少量辅助模块
- 仍以不破坏主干为前提

### 如果是“激进”
- 允许更明显的结构调整
- 但仍不得突破红线
- 必须写清楚变更理由和回退方案

## 3）关键段落解释

### “角色定位”

这一段的作用是**限制 agent 的人格和行为倾向**。
你这个项目不适合“自主发散型 agent”，更适合“保守型实现代理”。

### “项目背景”

这一段的作用是让 agent 明白：

* 什么是系统权威
* 什么不能碰
* 什么是核心原则

如果没有这段，agent 很容易把你的仓库当普通 Python 项目来改。

### “允许修改范围 / 禁止修改范围”

这是最关键的边界控制。
它直接决定 agent 会不会乱改核心主干。

### “成功标准”

很多 agent 会把“功能勉强能跑”当成功。
这段是告诉它：**你的项目更重视显式状态、审计、测试、重放。**

### “执行流程”

这是为了防止 agent 上来就乱改。
你要的是：

1. 先读
2. 再总结
3. 再计划
4. 再实现
5. 再测试
6. 再汇报

### “停止条件”

这是最重要的安全阀。
它保证任务如果越界，agent 会停下而不是硬干。
